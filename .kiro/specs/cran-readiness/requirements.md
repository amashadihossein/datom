# Requirements Document

## Introduction

This feature prepares the `datom` R package for its first submission to CRAN (the
Comprehensive R Archive Network). The package is currently pre-release (version
`0.0.0.9001`) and has not been published; there are therefore no backward-compatibility
or lifecycle concerns, and APIs, files, and metadata may be changed freely.

The work is driven by findings that were verified directly against the current codebase.
The findings fall into four clearly separated groups so that the design and tasks phases
can sequence them independently:

- **Blockers** — issues that would cause `R CMD check --as-cran` to emit an ERROR or
  WARNING, preventing acceptance. These are mechanical, low-risk metadata/namespace fixes.
- **Should-Fix** — issues that fall into the class CRAN reviewers commonly reject by hand
  even when `R CMD check` passes (e.g. `\dontrun{}` examples, missing `cran-comments.md`,
  unquoted software names). The examples workstream is the largest effort and is treated
  as a distinct workstream from the mechanical blockers.
- **Minor** — polish items that improve submission quality (license-holder consistency,
  a user-facing `NEWS.md`, optional author metadata).
- **Future Work** — explicitly deferred and NOT designed in this effort: enabling
  vignettes/examples to actually execute in CI via repository secrets. Captured here only
  as a high-level scope note because it is a likely remedy if CRAN raises it during review.

The authoritative validation gate for every requirement below is `R CMD check --as-cran`
(equivalently `devtools::check()`). **R is not available in this environment**, so all
verification criteria describe the expected outcome of running that gate; the gate itself
is run by the maintainer or CI, not assumed already-run here.

## Glossary

- **Package**: The `datom` R package being prepared for CRAN submission.
- **CRAN**: The Comprehensive R Archive Network, the target distribution repository.
- **CRAN_Check**: The validation gate `R CMD check --as-cran` (equivalently
  `devtools::check()`), run by the maintainer or CI. It is the source of truth for
  pass/fail of structural requirements.
- **CRAN_Policy**: The published CRAN Repository Policy and Writing R Extensions rules
  that the submission must satisfy.
- **DESCRIPTION_File**: The package `DESCRIPTION` file declaring metadata and dependencies.
- **NAMESPACE_File**: The package `NAMESPACE` file, generated by `roxygen2`.
- **Imports_Field**: The `Imports` field of the `DESCRIPTION_File` (hard runtime dependencies).
- **Suggests_Field**: The `Suggests` field of the `DESCRIPTION_File` (optional dependencies).
- **Null_Coalesce_Operator**: The `%||%` infix operator, used in 17 source files, sourced
  from `rlang`.
- **Exported_Function**: A function listed via `export(...)` in the `NAMESPACE_File`.
- **Example_Block**: The `@examples` section of an `Exported_Function`'s roxygen documentation.
- **Dontrun_Example**: An `Example_Block` (or portion) wrapped in `\dontrun{}`, which CRAN
  reviewers reject.
- **Donttest_Example**: An `Example_Block` (or portion) wrapped in `\donttest{}`, permitted
  by CRAN for examples that are runnable but too slow or environment-dependent for routine checks.
- **Runnable_Example**: An `Example_Block` that executes successfully under `CRAN_Check`
  without external network access or credentials.
- **Local_Backend**: The local filesystem storage backend (`datom_store_local()`), which,
  together with the bundled `inst/extdata`, enables `Runnable_Example`s for local-backend
  functions without network or credentials.
- **Cran_Comments_File**: The `cran-comments.md` file submitted alongside the Package
  summarizing test environments and check results.
- **News_File**: The `NEWS.md` file describing user-facing changes per release.
- **License_File**: The `LICENSE` file (CRAN `MIT + file LICENSE` form) naming the year and
  copyright holder.
- **License_Md_File**: The `LICENSE.md` file containing the full MIT license text.
- **Authors_Field**: The `Authors@R` field of the `DESCRIPTION_File`.
- **Maintainer**: The person or CI process that runs `CRAN_Check` and submits the Package.

## Requirements

---

## Group A — Blockers (would fail `R CMD check --as-cran`)

### Requirement 1: Declare `glue` and `yaml` as runtime dependencies

**User Story:** As the Maintainer, I want `glue` and `yaml` declared as hard dependencies,
so that core code paths that use them unconditionally do not fail when the optional packages
are absent.

#### Acceptance Criteria

1. THE DESCRIPTION_File SHALL list `glue` in the Imports_Field.
2. THE DESCRIPTION_File SHALL list `yaml` in the Imports_Field.
3. THE DESCRIPTION_File SHALL NOT list `glue` in the Suggests_Field.
4. THE DESCRIPTION_File SHALL NOT list `yaml` in the Suggests_Field.
5. THE DESCRIPTION_File SHALL retain `git2r` in the Suggests_Field.
6. THE DESCRIPTION_File SHALL retain `rio` in the Suggests_Field.
7. WHEN CRAN_Check is run, THE CRAN_Check SHALL report no ERROR or WARNING attributable to
   undeclared use of `glue` or `yaml`.

### Requirement 2: Declare `utils` as a runtime dependency

**User Story:** As the Maintainer, I want `utils` declared as a dependency, so that
`utils::` calls across the source resolve under check without an undeclared-import note.

#### Acceptance Criteria

1. THE DESCRIPTION_File SHALL list `utils` in the Imports_Field.
2. WHEN CRAN_Check is run, THE CRAN_Check SHALL report no ERROR, WARNING, or NOTE
   attributable to undeclared use of `utils` functions (including `packageVersion`,
   `modifyList`, `read.csv`, `glob2rx`, `head`, and `globalVariables`).

### Requirement 3: Resolve the `%||%` operator through an explicit import

**User Story:** As the Maintainer, I want the `%||%` operator imported from `rlang`, so that
the Package does not depend on an undeclared base-R floor for an operator used in 17 files.

#### Acceptance Criteria

1. THE NAMESPACE_File SHALL import the Null_Coalesce_Operator from `rlang`.
2. THE NAMESPACE_File SHALL be regenerated by `roxygen2` rather than edited by hand.
3. THE DESCRIPTION_File SHALL retain `rlang` in the Imports_Field.
4. WHEN CRAN_Check is run, THE CRAN_Check SHALL report no ERROR, WARNING, or NOTE
   attributable to an undefined or unimported Null_Coalesce_Operator.
5. WHERE the Null_Coalesce_Operator is resolved via the `rlang` import, THE Package SHALL NOT
   declare a base-R version floor solely to provide that operator.

## Group B — Should-Fix (reviewer rejection class)

### Requirement 4: Set the first-submission version

**User Story:** As the Maintainer, I want the version set to a release number, so that the
submission presents as a first stable release rather than a development build.

#### Acceptance Criteria

1. THE DESCRIPTION_File SHALL set the `Version` field to `0.1.0`.
2. THE DESCRIPTION_File SHALL NOT carry a four-component development version suffix
   (the `.9000`-style suffix) for the submission.

### Requirement 5: Convert examples to be acceptable to CRAN reviewers

**User Story:** As a Maintainer, I want examples that run rather than `\dontrun{}` blocks, so
that reviewers can execute documented usage and do not reject the submission on examples.

#### Acceptance Criteria

1. THE Package SHALL NOT contain any Dontrun_Example in the documentation of an
   Exported_Function.
2. WHERE an Exported_Function operates against the Local_Backend, THE Example_Block for that
   function SHALL be a Runnable_Example using the bundled `inst/extdata` and
   `datom_store_local()` without network access or credentials.
3. WHERE an Exported_Function requires network access, credentials, or other resources
   unavailable under CRAN_Check, THE Example_Block for that function SHALL use a
   Donttest_Example rather than a Dontrun_Example.
4. THE Package SHALL provide an Example_Block for each Exported_Function.
5. WHEN CRAN_Check is run, THE CRAN_Check SHALL report no ERROR or WARNING arising from
   example execution.

### Requirement 6: Provide `cran-comments.md`

**User Story:** As the Maintainer, I want a `cran-comments.md` file, so that reviewers see the
test environments and check results for the submission.

#### Acceptance Criteria

1. THE Package SHALL include a Cran_Comments_File at the package root.
2. THE Cran_Comments_File SHALL state the R versions and platforms against which CRAN_Check
   was run.
3. THE Cran_Comments_File SHALL state the result of CRAN_Check, including the count and text
   of any remaining NOTE.
4. THE Cran_Comments_File SHALL state the downstream reverse-dependency check status (none
   for a first submission).

### Requirement 7: Quote software names in `DESCRIPTION` per CRAN policy

**User Story:** As the Maintainer, I want software and service names quoted in the Title and
Description, so that the metadata complies with CRAN_Policy.

#### Acceptance Criteria

1. THE DESCRIPTION_File SHALL enclose the software/service name `git` in single quotes
   within the `Title` and `Description` fields wherever the name appears.
2. THE DESCRIPTION_File SHALL enclose the software/service name `GitHub` in single quotes
   within the `Title` and `Description` fields wherever the name appears.
3. THE DESCRIPTION_File SHALL enclose the software/service name `S3` in single quotes within
   the `Title` and `Description` fields wherever the name appears.
4. WHEN CRAN_Check is run, THE CRAN_Check SHALL report no NOTE requesting that software names
   be quoted in the Title or Description.

## Group C — Minor

### Requirement 8: Make the license holder consistent

**User Story:** As the Maintainer, I want one consistent copyright holder across license
files and author metadata, so that the license presentation is unambiguous.

#### Acceptance Criteria

1. THE License_File and THE License_Md_File SHALL name the same copyright holder.
2. THE copyright holder named in THE License_File SHALL match the author identity in the
   Authors_Field of the DESCRIPTION_File.
3. WHEN CRAN_Check is run, THE CRAN_Check SHALL report no WARNING or NOTE attributable to
   an inconsistent or malformed license declaration.

### Requirement 9: Rewrite `NEWS.md` as user-facing release notes

**User Story:** As a new user, I want release notes that describe what the released version
does in user-facing terms, so that I am not shown stale internal development-phase narrative.

#### Acceptance Criteria

1. THE News_File SHALL contain a top entry for version `0.1.0`.
2. THE News_File SHALL describe the `0.1.0` released state in user-facing terms.
3. THE News_File SHALL NOT reference internal development phase numbers (for example
   "Phase 15", "Phase 16", "Phase 17").
4. THE News_File SHALL NOT reference functions that are absent from the Package source
   (for example `datom_init_gov`, `datom_pull_gov`, `datom_decommission`).

### Requirement 10: Optional author metadata enrichment

**User Story:** As the Maintainer, I want optional author and citation metadata, so that
attribution and citation are well-formed for a first release.

#### Acceptance Criteria

1. WHERE the author elects to publish an ORCID, THE Authors_Field SHALL record the ORCID
   identifier in the `comment` field of the corresponding `person` entry.
2. WHERE the author elects to provide citation metadata, THE Package SHALL include an
   `inst/CITATION` file.
3. WHEN CRAN_Check is run, THE CRAN_Check SHALL report no WARNING or NOTE attributable to a
   malformed Authors_Field or `inst/CITATION` file.

## Group D — Future Work (deferred — NOT designed in this effort)

### Requirement 11: CI-executable vignettes and examples (deferred)

**User Story:** As the Maintainer, I want a documented high-level scope for making some
vignette and example steps actually execute in CI, so that reproducible documentation can
run rather than being marked `eval = FALSE`, should CRAN request it during review.

> **Scope note (deferred):** This requirement is intentionally captured at a high level and
> is OUT OF SCOPE for the current design and tasks. It is recorded because it is a likely
> remedy IF CRAN raises documentation-execution concerns during submission. A future effort
> would: (a) identify which flows need live execution (`S3`-backed and `GitHub`-backed) and
> which can stay on the Local_Backend; (b) define the repository secrets / environment
> variables required (e.g. cloud storage credentials and a `GitHub` token) and how they are
> injected into GitHub Actions without being committed or printed, consistent with the
> Package's explicit-secret-handling principle; (c) decide the partial-vs-all-or-none
> tradeoff — the Maintainer accepts enabling SOME vignettes/examples to run rather than all;
> and (d) document reproducibility implications of network/credentialed steps in CI. This
> requirement SHALL NOT be designed or implemented as part of the current effort.

#### Acceptance Criteria

1. THE Package documentation SHALL record the high-level scope described in the scope note
   above as deferred future work.
2. THE current design and tasks phases SHALL NOT include implementation of CI-executable
   vignettes or credentialed example execution.
